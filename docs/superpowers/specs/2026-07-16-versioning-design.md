# Версионность и update.sh

Дата: 2026-07-16

## Проблема

Скрипты проекта (`usage-monitor.sh`, `statusline-with-limits.sh`, `notify-attention.sh`) и launchd-plist копируются в `~/.claude/` один раз, при запуске `install.sh`. Когда в репозитории появляется фикс (как недавний фикс тихих сбоев опроса в `usage-monitor.sh`), установленная копия не обновляется сама — нужно вручную копировать файл или перезапускать `install.sh` целиком (что заново гоняет всю логику установки хуков и launchd). Нет и способа узнать, на какой версии проекта установленная копия находится, и есть ли новее.

## Цель

1. Дать проекту версионность (номер + changelog), видимую пользователю.
2. Дать отдельный лёгкий скрипт обновления, который синхронизирует установленные файлы с текущим состоянием локальной копии репозитория (без git pull внутри себя — это делает пользователь), не трогая то, что пользователь не устанавливал.

## Компоненты

### `VERSION`

Один файл в корне репозитория, одна строка — semver, например `0.2.0`. Источник истины для `install.sh` и `update.sh`.

Стартовые версии:
- `0.1.0` — всё, что уже смержено на момент этой задачи (алерты по лимитам, «needs your attention», фикс jitter при сбросе окна, фикс тихих сбоев опроса).
- `0.2.0` — эта задача (версионность + `update.sh`).

### `CHANGELOG.md`

Формат [Keep a Changelog](https://keepachangelog.com/), на русском (проект уже русскоязычный). Секции `## [0.1.0] - 2026-07-16` и `## [0.2.0] - 2026-07-16`, каждая с подпунктами Added/Fixed, коротко описывающими, что вошло.

### `~/.claude/scripts/.limit-alerts-version`

Скрытый файл с установленной версией (содержимое — просто номер версии, без переносов строк). Пишется:
- `install.sh` — после копирования скриптов, значением из `VERSION`.
- `update.sh` — после успешного обновления, новым значением из `VERSION`.

Удаляется `uninstall.sh` вместе с остальными файлами проекта.

Отсутствие файла (для уже существующих установок, сделанных до этой фичи — как у текущего пользователя) трактуется `update.sh` как «версия неизвестна / старее любой известной» — апдейт применяется, не требуя полного `install.sh`.

### `lib/hooks.sh`

Новый файл в репозитории (не копируется в `~/.claude/`), содержит:
- `resolve_jq()` — те же 3 строки поиска `jq`, что сейчас в начале `install.sh`.
- `add_hook()` — функция идемпотентной регистрации хука в `settings.json` (сейчас определена внутри `install.sh`), без изменений в логике.
- `register_monitor_hooks()` — регистрирует Stop/SessionStart хуки на `usage-monitor.sh hook`.
- `register_attention_hooks()` — регистрирует Notification/Stop хуки на `notify-attention.sh`.

`install.sh` подключает (`source`) этот файл вместо инлайновых определений; поведение `install.sh` не меняется, только источник функций.

### `update.sh`

Новый файл в корне репозитория, рядом с `install.sh`/`uninstall.sh`.

Флаги:
- `--dry-run` — показать, что будет сделано, ничего не менять.

Алгоритм:
1. Проверка macOS + наличие `jq` (как в `install.sh`).
2. `REPO_VERSION=$(cat VERSION)`, `INSTALLED_VERSION=$(cat ~/.claude/scripts/.limit-alerts-version 2>/dev/null || echo "неизвестна")`.
3. Если версии совпадают — вывести `Уже установлена последняя версия (vX.Y.Z)` и выйти с кодом 0.
4. Для каждого управляемого файла (`usage-monitor.sh`, `statusline-with-limits.sh`, `notify-attention.sh`): если файл **уже существует** в `~/.claude/scripts/` — скопировать поверх свежую версию из репозитория (`chmod +x`); если не существует — пропустить (пользователь не ставил эту фичу при установке, `update.sh` не должен её включать самовольно).
5. launchd: если `~/Library/LaunchAgents/com.claude.usage-monitor.plist` не существует — пропустить шаг (пользователь ставил с `--no-launchd`, аналогично п.4). Если существует — сгенерировать plist из `launchd/com.claude.usage-monitor.plist.template` во временный файл и сравнить с уже установленным; если содержимое отличается — записать новый и сделать `launchctl bootout` + `bootstrap`.
6. Хуки: `source lib/hooks.sh`; всегда `register_monitor_hooks`; `register_attention_hooks` — только если `notify-attention.sh` присутствует в `~/.claude/scripts/` (после шага 4).
7. Записать `REPO_VERSION` в `.limit-alerts-version`.
8. Вывести из `CHANGELOG.md` все секции строго между `INSTALLED_VERSION` (не включая) и `REPO_VERSION` (включая) — если `INSTALLED_VERSION` неизвестна, вывести только секцию `REPO_VERSION`.

В режиме `--dry-run` шаги 4–7 не выполняют запись, только печатают, что было бы сделано (какие файлы обновлены, будет ли перезагружен launchd, какие хуки будут дозаявлены), и не трогают `.limit-alerts-version`.

### `install.sh`

Изменения:
- Использовать `lib/hooks.sh` для `add_hook`/регистрации хуков вместо инлайновой логики.
- После копирования скриптов записать `$SCRIPTS_DIR/.limit-alerts-version` = содержимое `VERSION`.

### `usage-monitor.sh`

Первая строка вывода режима `status` — `claude-code-limit-alerts vX.Y.Z` (читает `~/.claude/scripts/.limit-alerts-version`; если файла нет — строку не печатать, чтобы не ломать вид у тех, кто ещё не обновился).

### `README.md` / `README.en.md`

Короткая секция «Обновление» между «Установка» и «Настройка»: `./update.sh` (и `--dry-run`), ссылка на `CHANGELOG.md`.

## Данные и поток

```
git pull (вручную, пользователем)
        │
        ▼
   ./update.sh
        │
   VERSION (repo) vs .limit-alerts-version (~/.claude/scripts)
        │
   равны? ──yes──> "уже актуально", exit 0
        │no
        ▼
   copy managed *.sh что уже стоят  →  chmod +x
        │
   regen plist → diff → bootout+bootstrap если изменился
        │
   source lib/hooks.sh → идемпотентная регистрация хуков
        │
   write .limit-alerts-version = REPO_VERSION
        │
   print CHANGELOG.md diff (installed..repo]
```

## Обработка ошибок

- Не macOS / нет `jq` — как в `install.sh`, понятное сообщение в stderr, `exit 1`.
- `~/.claude/scripts/` не существует вовсе (проект никогда не ставился) — `update.sh` печатает «Похоже, проект не установлен — запустите `./install.sh`» и выходит с `exit 1` (ничего не создаёт молча).
- launchd `bootout`/`bootstrap` — как в `install.sh`, ошибки `bootout` подавляются (`|| true`), т.к. агент может быть не загружен.
- Отсутствующий `CHANGELOG.md`-раздел для версии (рассинхрон) — не считается фатальной ошибкой, просто ничего не печатается в шаге 8.

## Тестирование

Автотестов в проекте нет (чистый bash, без test-раннера) — так же, как для прошлого фикса, проверка вручную:

1. `bash -n install.sh update.sh` — синтаксис.
2. Симулировать «версия неизвестна»: удалить `.limit-alerts-version`, запустить `update.sh --dry-run`, убедиться, что показывает предполагаемые изменения и не пишет файлы.
3. Запустить `update.sh` по-настоящему: проверить, что скрипты в `~/.claude/scripts/` совпадают с репозиторием (`diff`), `.limit-alerts-version` обновился, hooks в `settings.json` не задублировались (запустить `update.sh` второй раз подряд — должен вывести «уже актуально» и не менять `settings.json`).
4. Проверить, что `notify-attention.sh`, если он не был установлен (`--no-attention`), не появляется в `~/.claude/scripts/` после `update.sh` и attention-хуки не регистрируются.
5. `usage-monitor.sh status` показывает строку версии.
6. `uninstall.sh` по-прежнему удаляет всё, включая `.limit-alerts-version`.

## Вне рамок

- `update.sh` не делает `git pull`/`fetch` сам (решено в брейнсторминге — пользователь обновляет репозиторий сам).
- `update.sh` не устанавливает функции, которые не были установлены изначально (не включает `--no-attention`/`--no-statusline`/`--no-launchd` постфактум).
- Хэш-сверка файлов не используется — сравнение только по номеру версии.
